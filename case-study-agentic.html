Case Study 02 · Agent Control Plane

Agentic Control Plane

A bounded execution platform for goals, commitments, approvals, automations, memories, documents, and integrations.

Role
Product architect and principal engineer
Timeframe
2026 · Active
Source
View repository
01 / CHALLENGE

Designing for accountable autonomy

Long-running agent work crosses authentication, integrations, background execution, approvals, recovery, and auditability. A useful control plane must keep these concerns connected while preventing an autonomous workflow from silently escalating into a high-risk external action.

02 / ARCHITECTURE

A bounded path from intent to evidence

Requests and integration signals enter a governed API boundary, are classified for readiness and risk, and become durable jobs. The worker executes bounded work while the dashboard exposes commitments, approvals, evidence, recovery state, and learned context.

  1. Capture

    The Next.js API accepts goals, commitments, watcher events, and integration signals with actor context.

  2. Govern

    Policy and connector readiness classify risk, permissions, preconditions, and approval requirements.

  3. Enqueue

    Long-running activity becomes a durable worker job instead of remaining inside a request lifecycle.

  4. Execute

    A separate worker processes bounded tasks with concurrency controls, idempotency, and recovery paths.

  5. Evidence

    PostgreSQL-backed state records artifacts, audit events, approvals, provenance, and learned context.

03 / OWNERSHIP

What I owned and shaped

Product boundaries, system structure, delivery controls, and the evidence needed to operate the system responsibly.

Scope and execution

  • Designed the commitment-first product model and the bounded execution loop connecting dashboard, APIs, worker, and persistence.
  • Implemented governance boundaries for actor context, workspace roles, approvals, mutation preconditions, rate limits, and external side effects.
  • Structured integrations and preview capabilities around explicit readiness signals instead of presenting incomplete surfaces as production-ready.
  • Created operational runbooks for worker durability, recovery, observability, connector reconciliation, security response, and deployment.

Controls built into the system

  • Authenticated API routes with actor and workspace context
  • Role permissions, mutation preconditions, request identity, and abuse-rate controls
  • Approval gates for higher-risk external side effects
  • Idempotency and durable job execution for retry-safe workflows
  • Security headers, CSP, public-origin validation, and controlled sharing
  • Operational readiness, recovery, and connector reconciliation surfaces
04 / DECISIONS

Engineering choices that define the product

The differentiating decisions are less about framework selection and more about authority, state, failure behavior, and operational truth.

DECISION 01

Separate request from execution

Long-running work is queued for a dedicated worker, keeping API lifecycles bounded and making retries and recovery explicit.

DECISION 02

Approval before consequence

Higher-risk external actions require an approval boundary; connectors do not gain implicit authority from an agent request.

DECISION 03

Capability honesty

Feature readiness is tracked in code so preview integrations and incomplete workflows are not represented as fully operational.

DECISION 04

Durable operating context

Goals, commitments, artifacts, audit state, and memories are persisted as product state rather than hidden inside a transient model conversation.

05 / EVIDENCE

How the implementation is validated

Evidence is layered across deterministic tests, integration boundaries, operational checks, and documented limitations.

Validation coverage

  • Repository-level type, lint, unit, integration, and smoke gates across the monorepo
  • Security regression coverage for APIs, authentication, side effects, sharing, and connector boundaries
  • Capability smoke checks that fail when production claims lack required readiness evidence
  • Runbook-backed validation for worker durability, connector recovery, observability, and deployment
  • Audit matrices connecting controls to implementation and executable evidence

Delivered outcomes

  • A modular TypeScript control plane with a Next.js dashboard/API, separate worker, and PostgreSQL production state.
  • A coherent execution model connecting commitments, approvals, automations, documents, memories, integrations, and recovery.
  • Explicit product readiness boundaries that separate operational core workflows from preview capabilities.
  • A reusable governance approach for keeping human accountability inside autonomous and human-in-the-loop work.
06 / TRADEOFFS

Honest boundaries and the next iteration

The system is presented with its operating constraints intact. Those boundaries determine what should be strengthened next.

Limitations

  • Several integration, watcher, memory, workflow-template, and autopilot surfaces remain preview capabilities until their readiness gates are satisfied.
  • Production operation depends on correctly configured shared state, authentication, workers, and connector credentials.
  • The control plane coordinates work; it cannot remove the underlying reliability and authorization constraints of external providers.
  • Higher autonomy remains intentionally bounded by governance and approval requirements.

Next steps

  • Close capability blockers before promoting preview surfaces to operational status.
  • Strengthen end-to-end recovery evidence for connector and worker failure modes.
  • Continue reducing operator load through calibrated batching, recommendations, and traceability.
