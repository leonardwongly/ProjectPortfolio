Case Study 01 · Governed AI

AgentForge Merge Guard

A GitHub-first change-control system that makes high-risk and agent-assisted pull requests deterministic, reviewable, and auditable.

Role
Product architect and principal engineer
Timeframe
2026 · Active
Source
View repository
01 / CHALLENGE

Designing for accountable autonomy

Teams adopting AI-assisted development need stronger evidence and decision trails around risky changes. Detection alone is insufficient: governance must identify the applicable policy, required evidence, accountable reviewer, decision state, and policy version without delegating blocking authority to a probabilistic model.

02 / ARCHITECTURE

A bounded path from intent to evidence

A signed GitHub event enters a queue-backed evaluation pipeline. Deterministic packages derive facts, apply versioned policy, persist the decision record, and publish the result back to the pull request.

  1. Ingest

    Fastify verifies GitHub webhook signatures and records deliveries before evaluation.

  2. Queue

    BullMQ separates webhook acknowledgement from asynchronous pull-request evaluation.

  3. Evaluate

    Detectors extract verified facts; policy-as-code resolves evidence and reviewer requirements.

  4. Record

    PostgreSQL stores policy versions, evaluations, audit events, and Change Control Records.

  5. Decide

    GitHub checks expose pass, warning, block, or authorized override outcomes to humans.

03 / OWNERSHIP

What I owned and shaped

Product boundaries, system structure, delivery controls, and the evidence needed to operate the system responsibly.

Scope and execution

  • Defined the product boundary: deterministic governance rather than autonomous code approval.
  • Designed the TypeScript monorepo across API, dashboard, worker, policy, detector, evidence, reviewer, record, and security packages.
  • Established gradual adoption modes—observe, warn, enforce, and optimize—so policy can mature without an abrupt rollout.
  • Documented deployment, self-hosting, policy tuning, testing, security handling, and operational recovery paths.

Controls built into the system

  • Signed GitHub webhook verification and duplicate-delivery handling
  • Policy schema validation, applicability scoping, and versioned policy packs
  • Role-constrained overrides with required reason, visibility, and audit trail
  • Secret redaction and source-excluding exports
  • Production readiness checks for database, queue, authentication, and proxy trust
  • Human approval retained for risk acceptance
04 / DECISIONS

Engineering choices that define the product

The differentiating decisions are less about framework selection and more about authority, state, failure behavior, and operational truth.

DECISION 01

Deterministic authority

Blocking outcomes come from explicit rules and verified facts. Optional AI output is disabled by default and advisory only.

DECISION 02

Durable change records

Each governed pull request produces an authoritative Change Control Record tied to the policy version, evidence, reviewers, checks, and overrides.

DECISION 03

Metadata-first storage

The system excludes source-code storage by default, redacts secrets, and keeps exported records focused on governance evidence.

DECISION 04

Fail-closed production

Signed webhooks, trusted actor identity, session controls, and storage policy are explicit readiness requirements rather than permissive fallbacks.

05 / EVIDENCE

How the implementation is validated

Evidence is layered across deterministic tests, integration boundaries, operational checks, and documented limitations.

Validation coverage

  • Unit coverage for policy parsing, detectors, evidence, reviewer routing, redaction, overrides, and records
  • Integration coverage for webhook ingestion, policy preview, evidence updates, overrides, and exports
  • Fixture scenarios covering sensitive paths, CI changes, test weakening, dependencies, migrations, secrets, and overrides
  • Playwright end-to-end coverage for dashboard rendering and primary workflows
  • Performance sanity checks for large changed-file lists and repeated synchronize events

Delivered outcomes

  • A working self-hosted V1 spanning API, dashboard, asynchronous worker, PostgreSQL persistence, and GitHub check publication.
  • An explainable decision trail that answers what triggered, what is missing, who must review, and which policy version applied.
  • A controlled adoption path from visibility-only observation to enforceable merge protection.
  • A reusable governance model: deterministic checks decide, AI explains, and humans approve risk.
06 / TRADEOFFS

Honest boundaries and the next iteration

The system is presented with its operating constraints intact. Those boundaries determine what should be strengthened next.

Limitations

  • Merge Guard reports whether configured requirements are satisfied; it does not certify that code is secure, correct, or compliant.
  • The current release is self-hosted rather than a managed SaaS offering.
  • AI-based blocking decisions and autonomous merge approval are intentionally outside the product boundary.
  • Production object-storage export adapters remain outside the current V1 scope.

Next steps

  • Validate positioning and policy scope with external platform and security teams.
  • Expand deployment evidence and operational telemetry for enforce-mode adoption.
  • Evolve enterprise packaging only after user evidence supports retention, SSO, and audit-export needs.
